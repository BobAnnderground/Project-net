# Fixnet — Software Requirements Specification (SRS)

**Версия:** 0.1 (черновик для прототипирования)
**Статус:** Прототип без интеграций — все данные и соединения мокаются
**Платформа:** Windows-приложение, окно 1200×800px

---

## 1. Введение

### 1.1 Назначение документа
Документ описывает функциональные требования, сущности, их атрибуты, состояния и взаимосвязи для приложения Fixnet — системы маршрутизации трафика на уровне отдельных сервисов (per-app routing), в отличие от классического VPN, работающего на уровне всего устройства.

Документ предназначен для генерации архитектуры и кода прототипа (без бэкенд-интеграций, без реального сетевого драйвера) с целью тестирования пользовательских флоу.

### 1.2 Область действия
Прототип реализует:
- UI/UX всех основных экранов;
- локальное (mock) состояние сущностей;
- имитацию сетевых событий (подключение, деградация, обрыв, переключение на аварийный мост);
- имитацию мониторинга в реальном времени (случайные/детерминированные mock-данные).

Прототип **не реализует**:
- реальный сетевой драйвер и перехват трафика;
- реальную аутентификацию/подписку (можно замокать статус);
- реальные серверы и маршруты;
- реальную поддержку/тикеты.

### 1.3 Термины
| Термин | Значение |
|---|---|
| Сервис (Service) | Приложение/игра/сайт, трафик которого может маршрутизироваться через Fixnet |
| Регион (Region) | Точка выхода трафика (например, Sweden, Germany) |
| Маршрут (Route) | Сетевой путь между устройством и целевым сервисом через инфраструктуру Fixnet |
| Соединение (Connection) | Активная сессия передачи трафика конкретного сервиса |
| Пресет (Preset) | Именованный набор настроек сервисов и регионов |
| Мост (Bridge) | Резервный способ доступа к инфраструктуре при недоступности основного маршрута |
| Библиотека (Library) | Каталог заранее подготовленных сервисов |

---

## 2. Общее описание системы

Fixnet не туннелирует весь трафик пользователя. Пользователь явно выбирает набор сервисов, для каждого из которых индивидуально настраивается регион выхода и параметры подключения. Всё остальное продолжает работать через обычное интернет-соединение напрямую.

Ключевая идея: **N сервисов → N независимых маршрутов**, работающих параллельно.

### 2.1 Основной жизненный цикл
1. Пользователь авторизуется (mock).
2. Проверяется подписка (mock-статус: активна/истекла).
3. Загружается библиотека сервисов (mock-каталог).
4. Пользователь добавляет сервисы в личную библиотеку.
5. Настраивает регион и параметры каждого сервиса.
6. Опционально формирует пресет.
7. Нажимает «Запустить» — активируется mock сетевой слой.
8. Приложение начинает «маршрутизировать» настроенные сервисы (симуляция).
9. Мониторинг отображает статус/задержку/стабильность каждого соединения.
10. При сбоях (симулируемых) показываются уведомления и предлагается аварийный мост.

## 3. Сущности, атрибуты, состояния

### 3.1 User (Пользователь)

**Атрибуты:**
| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| name | string | Отображаемое имя |
| email | string | Почта (mock) |
| subscriptionStatus | enum: `active`, `expired`, `trial` | Статус подписки |
| subscriptionExpiresAt | date | Дата окончания |
| serviceLibrary | Service[] | Личная библиотека сервисов пользователя |
| presets | Preset[] | Список пресетов |
| appSettings | AppSettings | Настройки приложения |
| bridges | Bridge[] | Список сконфигурированных аварийных мостов |

**Состояния подписки:** `active → expired`, `trial → active`, `trial → expired`.

---

### 3.2 Service (Сервис)

Главная сущность приложения.

**Атрибуты:**
| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| name | string | Название (ChatGPT, Discord, Steam...) |
| icon | string (url/asset) | Иконка |
| category | enum: `ai`, `game`, `streaming`, `browser`, `messenger`, `other` | Тип сервиса |
| detectionMethod | enum: `domain`, `exe`, `game` (`exe`+`domain`+rules), `manual` | Способ определения трафика |
| domains | string[] | Список доменов (для web-сервисов) |
| exePath | string \| null | Путь к exe (для программ/игр) |
| additionalRules | NetworkRule[] | Доп. сетевые правила (для игр/ручных сервисов) |
| region | string (Region.id) | Выбранный регион выхода |
| enabled | boolean | Включен/выключен |
| encryption | enum: `on`, `off` | Шифрование |
| transportType | enum: `auto`, `tcp`, `udp`, `wireguard-like` | Тип транспорта |
| dnsMode | enum: `default`, `custom` | Использование DNS |
| advancedSettings | object | Расширенные настройки (для опытных пользователей) |
| isCustom | boolean | true — если создан пользователем вручную (не из библиотеки) |
| status | enum: `inactive`, `connecting`, `connected`, `degraded`, `error` | Текущий статус соединения |
| addedFromLibrary | boolean | Добавлен из общей библиотеки или создан вручную |

**Состояния (жизненный цикл сервиса):**
```
inactive --(пользователь включает + Запуск приложения)--> connecting
connecting --(успех)--> connected
connecting --(таймаут/ошибка)--> error
connected --(деградация сети, mock-событие)--> degraded
degraded --(восстановление)--> connected
degraded --(потеря маршрута)--> error
error --(ручной retry / авто-retry)--> connecting
connected/degraded/error --(пользователь выключает сервис)--> inactive
```

**Правила определения трафика по категориям:**
- **Веб-сервис** → `detectionMethod: domain`, обязателен непустой `domains[]`.
- **Программа** → `detectionMethod: exe`, обязателен `exePath`.
- **Игра** → `detectionMethod: game`, требует `exePath` + `domains[]` + `additionalRules[]`.
- **Ручной сервис** → `detectionMethod: manual`, `isCustom: true`; при создании показывается предупреждение об ответственности пользователя за корректность настройки.

### 3.3 NetworkRule (Сетевое правило)
Вложенная сущность внутри Service (используется в играх и ручных сервисах).

| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| type | enum: `port`, `ipRange`, `protocol` | Тип правила |
| value | string | Значение (например, порт или диапазон IP) |
| direction | enum: `inbound`, `outbound`, `both` | Направление |

### 3.4 Region (Регион)

Не является управляемой пользователем сущностью напрямую, но нужна как справочник для выбора точки выхода.

| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор (`sweden`, `germany-1`, `moscow-gaming-node`...) |
| displayName | string | Отображаемое имя |
| country | string | Страна |
| serverLoad | number (0–100) | Mock-загрузка сервера, % |
| recommendedFor | ServiceCategory[] | Для каких категорий сервисов рекомендован |

---

### 3.5 Route (Маршрут)

Формируется автоматически при включении сервиса; связывает Service с Region и текущим состоянием передачи.

| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| serviceId | string | Ссылка на Service |
| regionId | string | Ссылка на Region |
| status | enum: `idle`, `active`, `degraded`, `unavailable` | Статус маршрута |
| latencyMs | number | Задержка (mock, обновляется периодически) |
| stability | number (0–100) | Индекс стабильности (mock) |
| usesBridge | boolean | Использует ли сейчас аварийный мост |

---

### 3.6 Connection (Соединение)

Активная сессия конкретного Route в моменте. По сути — снимок состояния маршрута, используемый для отображения в мониторинге и истории.

| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| routeId | string | Ссылка на Route |
| startedAt | datetime | Время начала |
| endedAt | datetime \| null | Время окончания (если завершено) |
| qualityHistory | {timestamp, latencyMs, stability}[] | История качества для графика |

---

### 3.7 Preset (Пресет)

| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| name | string | Название (например, «Gaming», «AI») |
| serviceConfigs | {serviceId, region, enabled, ...overrides}[] | Снимок настроек сервисов |
| isActive | boolean | Применён ли пресет сейчас |
| createdAt | date | Дата создания |

**Состояния:** `saved → applied → saved` (применение пресета перезаписывает текущие настройки сервисов; предыдущее активное состояние не сохраняется автоматически, если не было отдельно сохранено как пресет).

---

### 3.8 Bridge (Аварийный мост)

| Поле | Тип | Описание |
|---|---|---|
| id | string | Идентификатор |
| name | string | Название/тип моста |
| status | enum: `available`, `connecting`, `connected`, `failed` | Статус |
| triggeredBy | string (routeId) \| null | Какой маршрут инициировал подключение моста |
| isAuto | boolean | Подключается автоматически или по запросу пользователя |

**Логика активации:**
```
Route.status → unavailable
  → приложение пытается восстановить основной маршрут
    → неудача → предлагает Bridge (если isAuto=false) ИЛИ подключает автоматически (если isAuto=true)
      → Bridge.status: available → connecting → connected
        → повторная попытка Route → active (usesBridge=true)
```

## 4. Связи между сущностями (ER-обзор)

```
User 1───* Service            (личная библиотека)
User 1───* Preset
User 1───* Bridge
Service 1───1 Region           (текущий выбранный регион)
Service 1───* NetworkRule       (только для game/manual)
Service 1───1 Route             (активный маршрут, если enabled)
Route 1───* Connection          (история сессий)
Route *───1 Bridge              (опционально, если usesBridge)
Preset 1───* ServiceConfigSnapshot (ссылается на Service.id)
Notification *───1 Service      (опционально, relatedServiceId)
```

## 5. Функциональные требования (укрупнённо)

### 5.1 Библиотека сервисов
- FR-1: Приложение отображает предустановленный каталог сервисов с иконкой, категорией и рекомендованным регионом.
- FR-2: Пользователь может добавить сервис из библиотеки в личный список одним действием.
- FR-3: Пользователь может создать ручной сервис, задав правила самостоятельно; при сохранении показывается предупреждение об ответственности.
- FR-4: Добавленный сервис по умолчанию находится в состоянии `inactive` до запуска приложения.

### 5.2 Настройка сервиса
- FR-5: Для каждого сервиса независимо настраиваются: включен/выключен, регион, шифрование, транспорт, DNS, расширенные параметры.
- FR-6: Изменение региона одного сервиса не влияет на другие.

### 5.3 Запуск и маршрутизация
- FR-7: По нажатию «Запустить» все `enabled` сервисы переходят `inactive → connecting → connected` (с симулированной задержкой).
- FR-8: Остальной (не сконфигурированный) трафик считается «напрямую» и не отображается как управляемый.
- FR-9: Каждый активный сервис отображает независимый статус, регион, задержку и стабильность.

### 5.4 Мониторинг
- FR-10: Главный экран показывает сводную картину: количество активных сервисов, средняя задержка, есть ли проблемы.
- FR-11: На детальном экране сервиса отображается график качества соединения (mock-история).
- FR-12: Периодически (симулированный таймер) состояние маршрутов может меняться (`connected → degraded → connected`, либо `→ error`), генерируя уведомления.

### 5.5 Уведомления
- FR-13: При смене статуса маршрута на `degraded`/`unavailable`/`error` создаётся Notification соответствующего типа.
- FR-14: Уведомление может содержать действие: показать альтернативный маршрут, подключить мост, перейти к сервису.

### 5.6 Аварийные мосты
- FR-15: При недоступности основного маршрута приложение предлагает подключить мост (или подключает автоматически, если так настроено).
- FR-16: После подключения моста маршрут помечается `usesBridge: true`, статус пытается вернуться в `active`.

### 5.7 Пресеты
- FR-17: Пользователь может сохранить текущую конфигурацию сервисов как именованный пресет.
- FR-18: Применение пресета перезаписывает текущие настройки enabled/region у соответствующих сервисов.

### 5.8 Настройки приложения
- FR-19: Общие настройки (тема, автозапуск, DNS, поведение окна) не привязаны к сервисам и хранятся отдельно.

## 6. Пользовательские сценарии (User Flows) для прототипа

1. **Первый запуск** → авторизация (mock) → пустая библиотека → онбординг с предложением добавить сервисы.
2. **Добавление сервиса** → Библиотека → выбор сервиса → «Добавить» → появляется в личном списке в состоянии `inactive`.
3. **Настройка сервиса** → открыть карточку сервиса → выбрать регион, включить → сохранить.
4. **Запуск** → кнопка «Запустить» на главном экране → сервисы переходят в `connecting → connected` → отображается сводка.
5. **Деградация и уведомление** → симулированное ухудшение маршрута → появляется Notification → пользователь кликает «Подключить мост» → Bridge подключается → маршрут восстанавливается.
6. **Создание пресета** → текущая конфигурация → «Сохранить как пресет» → ввод названия → появляется в списке пресетов.
7. **Применение пресета** → выбор пресета из списка → подтверждение → конфигурация сервисов перезаписывается.
8. **Ручной сервис** → «Создать сервис» → заполнение полей → предупреждение об ответственности → сохранение.
9. **Настройки приложения** → изменение темы/автозапуска/DNS.

## 7. Требования к UI (для прототипа)

- Платформа: Windows desktop, фиксированное окно **1200×800px** (не resizable для прототипа, либо с минимальным допуском).
- Структура окна: боковая навигация (Dashboard, Библиотека, Пресеты, Настройки) + основная область контента.
- Главный экран (Dashboard): список добавленных сервисов с live-статусами, общая сводка, кнопка «Запустить/Остановить».
- Модальные окна: добавление сервиса из библиотеки, создание ручного сервиса, предложение подключить мост, сохранение пресета.
- Центр уведомлений: выпадающая панель или отдельная секция со списком Notification, с возможностью пометить прочитанным и выполнить действие.
- Никаких реальных сетевых вызовов — все статусы, задержки и события генерируются mock-логикой (таймеры, псевдослучайные изменения) для тестирования флоу.

---

## 8. Нефункциональные требования (для прототипа)

- NFR-1: Прототип работает полностью на клиенте, без бэкенда.
- NFR-2: Состояние может храниться в памяти (не обязательно персистентно между перезапусками) — если нужна персистентность, использовать локальное хранилище.
- NFR-3: Симуляция сетевых событий должна быть настраиваемой (частота, вероятность деградации) для удобства тестирования флоу.
- NFR-4: UI должен четко отличать состояния `connected` / `degraded` / `error` / `inactive` визуально (цвет, иконка).

---

## 9. Ограничения прототипа

- Нет реальной аутентификации, подписки, платежей.
- Нет реального сетевого драйвера / перехвата трафика.
- Нет реальных серверов/регионов — используется статичный справочник.
- Нет реальной службы поддержки — «отправить запрос на добавление сервиса» может быть просто UI-заглушкой.
